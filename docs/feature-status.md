# TiebaLite 功能现状清单

最后更新：2026-03-22

这份清单用于说明当前复活阶段里，哪些能力已经可以稳定对外，哪些能力仍然落后于现行贴吧体验，哪些功能仍应保持实验性，哪些能力暂时不应对外承诺。

当前默认策略：

- 主支持范围为 Android 10+
- 第一阶段目标是“公开浏览 + 核心账号”
- 主号不参与账号功能验证
- 未经过测试号闭环验证的账号功能，不进入默认稳定路径

## 入口能力审计

| 层级 | 入口 | 当前状态 | 现状说明 |
| --- | --- | --- | --- |
| 一级 | 首页 | 稳定 | 公开浏览主干，搜索、吧页、帖子页都从这里自然进入。 |
| 一级 | 动态 | 稳定壳 + 守卫子页 | 热榜、推荐属于公开浏览稳定路径；关注流属于账号态二级入口，只在登录后显示。 |
| 一级 | 我 | 稳定壳 | 作为账号、历史、设置容器稳定可见，但内部入口按能力分层；首页 / 动态 / 我 的高频入口不再把用户带进误导性半成品。 |
| 二级 | 全局搜索 | 稳定 | 吧、贴、用户搜索都属于公开浏览闭环。 |
| 二级 | 话题榜 / 话题详情 | 稳定 | 已接成“榜单到详情”闭环，不再停在列表入口。 |
| 二级 | 吧页 / 帖子页 / 图片 / 视频 / 外链 | 稳定 | 仍是当前复活阶段的核心主干。 |
| 二级 | 浏览历史 | 稳定 | 本地阅读辅助入口，不依赖账号闭环。 |
| 二级 | 我的收藏 | 核心账号守卫 | 仅完整登录时显示；即使被深链或历史入口打开，也会先显示明确守卫页而不是静默空白。 |
| 二级 | 通知中心 | 核心账号守卫 | 已接 capability + session health gate；用户页入口会直接引导到登录或账号管理，而不是先进入守卫页。 |
| 二级 | 账号管理 / 登录 | 核心账号守卫 | 设置页会显示账号能力和当前会话健康摘要，登录结果以 `SessionHealth` 为唯一判定源。 |
| 二级 | OKSign 设置 | 核心账号 + 实验性混合 | 手动签到属于核心账号，自动签到继续保持实验性。 |
| 二级 | 自动签到开关 | 实验性 / 默认关闭 | 需显式开启实验性功能，不纳入稳定默认路径。 |
| 二级 | 回复 / 回帖入口 | 实验性 / 默认关闭 | 受 capability、session health、强制风险提示三重约束。 |
| 二级 | 发帖入口 | 暂不承诺 | 当前没有需要优先恢复的现代稳定主路径。 |
| 二级 | 服务中心等账号附属 WebView 入口 | 账号态附属入口 | 仅完整登录后出现；命中账号附属网页时会先检查完整会话，不再让半登录状态卡在空白 WebView。 |

## 可稳定使用

- 阅读优先的信息架构：
  主导航已收口为 `首页 / 动态 / 我`；消息下沉到用户页次级入口，首页不再直接暴露一键签到，吧页已关注场景下也不再把签到按钮放在高可见位置。
- 公开浏览主线：
  首页、Explore、全局搜索、吧页、帖子页已经是当前复活主干。
- 论坛页基础面板：
  当前已形成 `最新 / 精华 / 吧内搜索` 的可扩展结构，不再是硬编码双 tab。
- 话题链路：
  话题榜可以进入，话题详情已经有独立路由与页面，不再只停在列表入口。
- 帖子页基础内容能力：
  图片、视频、外链、楼中楼、IP 属地等内容阅读能力已经纳入公开浏览稳定路径。
- 复活状态提示：
  设置页现在除了能力分级摘要，还会直接说明“公开浏览优先”的当前定位；关于页也同步显示同一口径的复活说明。
- 可见入口与能力状态对齐：
  设置页的账号管理与 OKSign 入口现在会直接显示核心账号 / 实验性摘要；`Explore` 页的关注流 tab 也会随登录态及时收口，不再在切号或退出后残留旧入口。
- 首页 / 动态页提示：
  首页和 `Explore` 页已经显式提示“公开浏览优先”的恢复范围，不再让未登录用户误以为必须先补齐账号功能才能正常使用主线浏览能力。
- 用户页导航收口：
  消息入口现在会直接把用户送去登录或账号管理；收藏和服务中心等完整会话能力不再在半登录状态下继续暴露。
- 守卫页空态收口：
  收藏页和账号附属 WebView 入口现在都会显式显示“去登录 / 账号管理”守卫页，不再以空白页或半死不活的接口失败作为默认反馈。
- WebView 与站内链接分流：
  Tieba 贴子/吧链接会继续优先回到原生页，`/mo/q/checkurl` 会先解析真实目标，再决定是原生跳转、留在 WebView，还是外跳到系统浏览器 / Custom Tabs。

## 已落后但可继续做

- 核心账号 Alpha 现在是“可验证、但还未宣称稳定”的状态：
  登录结果已经统一收口到 `SessionHealth`，通知页和手动签到在会话不完整时会明确拦截并给出提示，退出/切号也会同步清理未读和后台轮询；但这些链路仍待测试号闭环确认后，才能升级为“已验证稳定”。
- 论坛面板深度仍落后于当前贴吧网页：
  现阶段只有 `最新 / 精华 / 吧内搜索`，还没有补齐更接近网页的 `图片 / 视频 / 推荐` 等内容维度。
- 通知中心完整度仍有限：
  当前核心方向是 `回复我 / @我 / 赞过我`。其中“赞过我”已经改成 `agree_list / agreeme` 主字段优先，仅在字段缺失时才回退到历史兼容字段，并会显示兼容提示；在测试号样本收实前，仍不应宣传为完全稳定。
- 手动签到已具备保守前置反馈：
  完整 session 下会正常入队；未登录、Cookie 不完整、账号信息不完整时会立刻提示当前会话状态，而不是等 Worker 在后台静默失败。
- Android 平台基线仍停在 SDK 34：
  当前目标是先把现代化兼容层收口，再单独处理 target/compile SDK 升级。
- 自定义交互兼容层仍是过渡态：
  placeholder、图片渐变、bottom sheet、swipe threshold 已经集中到项目内部封装，但底层仍有旧 API 兼容代码，后续要继续向新版 Compose/Foundation 迁移。

## 保持实验性

- 自动签到：
  默认继续作为实验性能力，不作为稳定功能宣传。
- 回帖：
  仍受 capability 和 session health 双重保护；当前入口现在会强制先显示“风险自负”提示，没有测试号闭环前不应当作稳定路径。
- 发帖：
  仍然不纳入第一阶段稳定范围；当前仓库里也没有需要优先恢复的现代独立发帖主路径。
- 任何未完成真实账号验证的接口分支：
  继续留在实验性或隐藏路径，不默认放开。

## 暂不承诺

- 发帖/回帖完整恢复
- 现行贴吧网页的全量论坛内容维度对齐
- 使用主号完成登录、通知、签到等高风险闭环验证
- Android 15/16 上的最终 target SDK 升级结果

## 当前建议验证顺序

1. 先验证公开浏览链路：
   首页、热榜、搜索、吧页、帖子页、图片、视频、外链。
2. 再用测试号验证核心账号：
   登录、会话健康、通知字段样本、手动签到、退出/切号清理。
3. 最后再决定是否推进实验性能力：
   自动签到、回帖、发帖。

## 已知技术债

- **P1: 构建工具链锁定在 Android 14** — `compileSdk`/`targetSdk` = 34，未来升级到 SDK 35/36 可能引发 edge-to-edge 布局回归，需先审计 `ImmersionBar`/`fitsSystemWindows` 用法。
- **P1: "赞过我"通知解析仍是启发式兼容** — `agree_list`/`agreeme` 作为主字段，缺失时回退到历史字段并显示兼容提示；需要测试号样本确认真实字段后精简。
- **P1: 核心账号路径未经真实账号端到端验证** — 登录、通知、签到、退出/切号的守卫逻辑已就位，但仍待测试号闭环确认。
- **P1: 论坛维度落后于网页版** — 当前只有 `最新 / 精华 / 吧内搜索`，缺少 `图片 / 视频 / 推荐` 等内容维度 tab。
- **P2: cleartext traffic 全局开启** — manifest 仍声明 `usesCleartextTraffic="true"`，主要 API 已迁移 HTTPS，待确认是否可收紧。
- **P2: 旧 OEM 兼容 manifest flags** — `largeHeap`、`notch_support`、`EasyGoClient` 等仍在，需逐项验证后清理。
- **P2: 设备身份兼容层仍是 shim** — 已不泄露真实硬件 ID，但 `AID`/`CUID`/`android_id` 等字段仍以假值填充，需配合真实账号验证。
- **P3: 旧 ViewPager/Fragment 基类疑似死代码** — `BaseFragment` 无活跃子类，待确认后删除。
